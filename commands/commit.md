# 프로젝트 맞춤형 커밋 가이드

이 가이드는 `/commit` 명령어가 입력되었을 때 실행해야 하는 워크플로우를 정의합니다.

## 1. 사전 검증 (Pre-commit Checks)
커밋을 생성하기 전, 아래 명령어를 실행하여 코드의 안정성을 먼저 확인합니다.
- `pnpm lint`: 코드 스타일 및 컨벤션 준수 여부 확인
- `pnpm build`: 빌드 성공 여부 확인
- 오류 발생 시 사용자에게 수정을 권고하고, `--no-verify` 옵션이 있을 때만 생략합니다.

## 2. 스테이징 및 분석 프로세스
1. **자동 추가**: 현재 스테이징된 파일이 없다면 `git add .`를 수행합니다.
2. **변경 분석**: `git diff --cached`를 통해 변경 사항을 논리적으로 분석합니다.
3. **커밋 분할**: 서로 다른 논리적 작업(예: 기능 추가와 리팩토링)이 섞여 있다면 작업을 나누어 여러 번의 커밋으로 진행할 것을 제안합니다.

## 3. 커밋 메시지 규격 (Conventional Commits + Gitmoji)
모든 커밋 메시지는 `<이모지> <타입>: <설명>` 형식을 따르며, **영문 현재 시제/명령조**로 작성합니다.

| 타입 | 이모지 | 용도 |
| :--- | :---: | :--- |
| **feat** | ✨ | 새로운 기능 추가 |
| **fix** | 🐛 | 버그 수정 |
| **docs** | 📝 | 문서 수정 (README, 주석 등) |
| **style** | 💄 | 코드 포맷팅, UI 스타일 변경 (로직 수정 없음) |
| **refactor**| ♻️ | 코드 리팩토링 (기능 변화 없는 구조 개선) |
| **perf** | ⚡️ | 성능 개선 |
| **test** | ✅ | 테스트 추가 및 수정 |
| **chore** | 🔧 | 빌드 설정, 패키지 매니저, 기타 도구 수정 |
| **design** | 🎨 | 코드 구조 개선 (UI/UX 포함) |
| **types** | 🏷️ | 타입 정의 추가 및 수정 (TypeScript) |

## 4. 실전 예시
- `✨ feat: add responsive navigation bar`
- `🐛 fix: resolve null pointer error in user profile`
- `♻️ refactor: simplify authentication logic`
- `🏷️ feat: add interface for API response`

## 5. 실행 규칙
- 커밋 제목은 **72자 이내**로 간결하게 작성합니다.
- 본문이 필요한 경우 제목과 한 줄 띄우고 상세 내용을 기술합니다.
- 사용자가 `/commit`을 입력하면 위 과정을 자동으로 수행하고 최종 메시지를 제안합니다.
